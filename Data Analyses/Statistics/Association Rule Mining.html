Chapter — Unsupervised Learning
Association Rule Mining
One-word definition Relationships
Association Rule Mining is an unsupervised machine learning technique that discovers statistically significant co-occurrence relationships between items in large transactional datasets, generating IF–THEN rules that reveal hidden purchase patterns, behaviours, or dependencies without labelled supervision.
Real-World Analogy — Market Basket Analysis
Transaction Data
A supermarket stores every receipt — each receipt is a transaction listing items bought together.
The Rule Found
{Bread, Butter} → {Milk}
"Customers who buy Bread & Butter also tend to buy Milk."
Business Value
Place Milk near Bread & Butter. Bundle them. Recommend them online. Optimise shelf layout.
Core Metrics — Used Across All ARM Algorithms
Support
How frequently the itemset appears in all transactions. Measures raw popularity.
Support(A) = freq(A) / N
N = total transactions. Range: 0 → 1.
Confidence
Given A is bought, how often is B also bought? Measures rule reliability.
Conf(A→B) = Support(A∪B) / Support(A)
Range: 0 → 1. High = reliable rule.
Lift
Are A and B truly correlated, or just individually popular? Corrects for base rate.
Lift(A→B) = Conf(A→B) / Support(B)
Lift > 1 = positive correlation. = 1 → independent. < 1 → negative.
min_support threshold: The minimum support value an itemset must meet to be considered frequent. Itemsets below this are discarded (pruned). Setting it too low → explosion of rules.
min_confidence threshold: The minimum confidence a rule must meet to be retained. Rules below this are too unreliable to act on. Both thresholds must be tuned for your dataset.
Three Types of Association Rule Mining
Type 01
Frequent Itemset Mining
Find all sets of items that appear together frequently. Generate rules from them. Classic approach.
Algorithm covered → Apriori
Type 02
Tree-Based Mining
Compress the transaction database into a compact prefix tree, then mine it without re-scanning data.
Algorithm covered → FP-Growth
Type 03
Sequential Pattern Mining
Items have a temporal order. Find patterns in sequences of events over time (e.g. customer journeys).
Algorithm covered → GSP
Type 01 · Frequent Itemset Mining
Apriori Algorithm
Agrawal & Srikant, 1994 · The foundational ARM algorithm
Apriori iteratively generates candidate itemsets of increasing size, pruning those that fail the minimum support threshold — exploiting the Apriori Property: any subset of a frequent itemset must also be frequent.
⚡ The Apriori Property (Anti-Monotone):
If itemset {A, B} is infrequent, then any superset {A, B, C} must also be infrequent. This lets us prune entire branches of the search space without evaluating them — this is the core insight that makes Apriori practical.
Key Terms
Transaction (T): A single record containing a set of items. E.g., one grocery receipt = {Bread, Milk, Eggs}.
Itemset: Any subset of items. A 1-itemset = {Bread}, 2-itemset = {Bread, Milk}, etc.
Frequent Itemset (L_k): An itemset whose support ≥ min_support. The goal is to find all of these.
Candidate Itemset (C_k): A k-sized itemset generated as a potential frequent itemset, before support is verified.
Pruning: Discarding candidate itemsets whose subsets are already infrequent — using the Apriori Property.
Association Rule (A→B): A→B means "when A occurs, B tends to occur." A = antecedent, B = consequent.
Algorithm Steps
01
Scan database → find frequent 1-itemsets (L₁)
Count occurrences of every single item. Keep only those with support ≥ min_support.
02
Generate candidates C_{k+1} from L_k (Join Step)
Join pairs of k-frequent itemsets that share a (k−1) prefix to generate (k+1)-candidate itemsets.
03
Prune C_{k+1} (Prune Step)
Remove any candidate whose (k)-subsets are NOT in L_k. These cannot be frequent by the Apriori Property.
04
Scan database → compute support for survivors
Count actual occurrences of each surviving candidate in the full transaction database.
05
Keep those with support ≥ min_support → L_{k+1}
Survivors become the new frequent itemset list for the next iteration.
06
Repeat steps 2–5 until no new frequent itemsets
When C_k = ∅ or L_k = ∅, stop. All frequent itemsets of all sizes are now found.
07
Generate rules from all frequent itemsets
For each frequent itemset, try all possible A→B splits. Keep rules where confidence ≥ min_confidence.
Bottleneck: Apriori scans the entire database once per itemset size k. For large datasets or low min_support, this creates massive I/O overhead and memory pressure — motivating FP-Growth.
Code Syntax (mlxtend)
apriori.py 
# ── 1. Import ─────────────────────────────
from mlxtend.frequent_patterns import apriori
from mlxtend.frequent_patterns import association_rules
from mlxtend.preprocessing   import TransactionEncoder
import pandas as pd

# ── 2. Prepare data ───────────────────────
dataset = [
    ['Bread', 'Milk'],
    ['Bread', 'Butter', 'Milk'],
    ['Butter', 'Milk', 'Eggs'],
    ['Bread', 'Butter'],
]
te    = TransactionEncoder()
te_df = pd.DataFrame(
    te.fit_transform(dataset),
    columns=te.columns_
)  # one-hot encoded boolean DataFrame

# ── 3. Find frequent itemsets ────────────
freq_items = apriori(
    te_df,
    min_support=0.5,     # 50% of transactions
    use_colnames=True
)

# ── 4. Generate association rules ────────
rules = association_rules(
    freq_items,
    metric="confidence",
    min_threshold=0.7
)

# ── 5. Get values ─────────────────────────
rules[['antecedents', 'consequents',
       'support', 'confidence', 'lift']]
Output columns explained
ColumnWhat it tells you
antecedentsThe IF part of the rule
consequentsThe THEN part of the rule

Type 02 · Tree-Based Mining
FP-Growth Algorithm
Han, Pei & Yin, 2000 · Frequent Pattern Growth
FP-Growth eliminates the costly repeated database scanning of Apriori by compressing all transaction data into a single compact FP-Tree structure, then mining frequent patterns directly from this tree using recursive conditional pattern base extraction.
⚡ Key Advantage over Apriori: Apriori scans the database once per itemset size. FP-Growth scans the database exactly twice — once to count item frequencies, once to build the FP-Tree. Everything after that works on the in-memory tree. Result: orders of magnitude faster on large datasets.
Key Terms
FP-Tree (Frequent Pattern Tree): A prefix tree where each path represents a transaction, each node stores an item name and count. Shared prefixes are merged, massively compressing the data.
Header Table: A table listing each frequent item with its total count and a pointer to the first occurrence in the FP-Tree. Used to traverse all nodes of an item quickly.
Conditional Pattern Base: For a given item X, the set of prefix paths in the FP-Tree that end at X, together with their counts. These are the contexts in which X appears.
Conditional FP-Tree: A smaller FP-Tree built from the conditional pattern base of item X. Mining it gives all frequent itemsets that contain X. Recursively applied.
Node-link: A linked list connecting all occurrences of the same item across different branches of the FP-Tree. Allows header table traversals without full tree scans.
Prefix Path: The path from the root to a specific node in the FP-Tree. Each prefix path represents a co-occurrence context for the item at that node.
Algorithm Steps
01
Scan DB once → count item frequencies
Remove items below min_support. Sort remaining items by frequency (descending) — this order is critical for tree compression.
02
Build the FP-Tree (second scan)
For each transaction, insert items (sorted by frequency) into the tree. Merge paths that share a common prefix, incrementing node counts.
03
Build Header Table with node-links
Create a lookup table of all frequent items → their total count → pointer to first tree node.
04
Mine tree starting from least frequent item
For each item (bottom of header table first), collect its conditional pattern base — all prefix paths leading to it.
05
Build Conditional FP-Tree
Filter the conditional pattern base (min_support), build a new smaller FP-Tree for this item's context.
06
Recursively mine conditional FP-Trees
Repeat steps 4–5 on each conditional tree until the tree is empty or has a single path. Single paths = enumerate all subsets directly.
Memory note: FP-Tree can still be huge if min_support is very low or data is very diverse. In those cases, consider disk-based variants or ECLAT.
Code Syntax (mlxtend)
fp_growth.py 
# ── 1. Import ─────────────────────────────
from mlxtend.frequent_patterns import fpgrowth
from mlxtend.frequent_patterns import association_rules
from mlxtend.preprocessing   import TransactionEncoder
import pandas as pd

# ── 2. Prepare data (same as Apriori) ────
te    = TransactionEncoder()
te_df = pd.DataFrame(
    te.fit_transform(dataset),
    columns=te.columns_
)

# ── 3. Find frequent itemsets ────────────
freq_items = fpgrowth(
    te_df,
    min_support=0.5,
    use_colnames=True
)   # much faster than apriori()

# ── 4. Generate rules ─────────────────────
rules = association_rules(
    freq_items,
    metric="lift",
    min_threshold=1.2
)

# ── 5. Sort by lift descending ────────────
rules.sort_values('lift', ascending=False)

# ── Alternative: mlxtend FPTree directly ──
from mlxtend.frequent_patterns import fpmax
# fpmax() → maximal frequent itemsets only

Reading the conditional pattern base:
Each row = a prefix path that leads to node m in the FP-Tree.
The count = how many transactions share that exact path ending at m.

From this base, we build the Conditional FP-Tree for m by counting how often each item appears across all prefix paths (weighted by count). Items below min_support are pruned.
Type 03 · Sequential Pattern Mining
GSP Algorithm
Generalized Sequential Patterns · Srikant & Agrawal, 1996
GSP extends association rule mining to ordered sequences of events, discovering patterns where items must appear in a specific temporal order across transactions, enabling analysis of customer journeys, web clickstreams, and time-series behaviours.
🕐 Key Distinction — Order matters in GSP:
In Apriori/FP-Growth, {Bread, Milk} and {Milk, Bread} are the same itemset. In GSP, <{Bread} → {Milk}> and <{Milk} → {Bread}> are different sequences — the temporal ordering defines a new dimension of meaning.
Key Terms
Sequence (S): An ordered list of itemsets (events), separated by time. Written as <{A}{B,C}{D}>. Each {} is one time-step.
Event / Element: A single time-step in a sequence. Can contain multiple items if they occur simultaneously at that time. E.g., {Bread, Butter} as one event.
Sequential Support: The fraction of all customer sequences that contain a given pattern as a sub-sequence (in order). Must be ≥ min_support.
Sub-sequence: Pattern <A,B,C> is a sub-sequence of <X,A,Y,B,Z,C> if A, B, C appear in that order (gaps allowed).
Window Size (maxspan): A time constraint — the pattern must occur within a specified time window. E.g., "all events within 7 days."
Gap Constraints (mingap/maxgap): Minimum and maximum allowed time between consecutive events in the pattern. Controls how "close" events must be.
Algorithm Steps
01
Sort & map sequences
Sort the database by customer ID and time. Map items to integer IDs. Group transactions into per-customer sequences.
02
Find frequent 1-sequences (F₁)
Count each single-item sequence. Keep only those with sequential support ≥ min_support.
03
Generate candidate k-sequences (C_k) — Join
Join pairs of (k−1) frequent sequences by appending one item/event. Three join types: append as new event, merge into existing event, or extend.
04
Prune C_k using contiguous sub-sequence check
Remove candidate sequences whose (k−1) contiguous sub-sequences are NOT all in F_{k−1}. Uses same anti-monotone property as Apriori.
05
Scan DB → count support for each candidate
Apply gap and window constraints while checking each candidate against actual customer sequences.
06
Repeat until no new frequent sequences
Keep extending. Stop when no candidates survive. The union of all F_k is the complete set of sequential patterns.
Use cases: Web page clickstreams · Customer purchase journey analysis · Medical event sequences · Fraud detection chains · Log file pattern mining.
Code Syntax (prefixspan)
gsp.py 
# ── 1. Import ─────────────────────────────
# pip install prefixspan
from prefixspan import PrefixSpan

# ── 2. Prepare sequential data ────────────
# Each inner list = one customer's sequence
# Each sub-list   = one timestep/event
sequences = [
    [['a', 'b'], ['c'], ['d']],   # customer 1
    [['a'], ['c'], ['b', 'd']],   # customer 2
    [['a', 'b'], ['d']],         # customer 3
    [['b'], ['c'], ['d']],       # customer 4
]

# ── 3. Initialise & train ────────────────
ps = PrefixSpan(sequences)

# ── 4. Get frequent sequences ────────────
# min_support as absolute count
results = ps.frequent(2)  # min_support = 2

# ── 5. Output: (support, pattern) ────────
for support, pattern in results:
    print(f"Support: {support}  Pattern: {pattern}")

# ── GSP with constraints (manual) ────────
# For gap/window constraints, use:
# pip install sequential-rules
from spmf import Spmf  # Java-backed, full GSP
alg = Spmf("GSP", arguments=[0.5])
alg.run(input_file)
patterns = alg.to_pandas_dataframe()

Decision Guide
Items in a basket, no time?
Use Apriori to learn. Use FP-Growth in production.
Large dataset, speed critical?
Always use FP-Growth. Far fewer DB scans.
Events ordered over time?
Use GSP. Web clickstreams, medical events, user journeys.
