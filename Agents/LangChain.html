Section 01

Introduction to LangChain

What is LangChain?

Imagine you want to build an AI application — not just a chatbot that answers a single question, but something that can read documents, browse the web, remember past conversations, and take actions like running code or sending emails. Doing this from scratch would require hundreds of lines of complex code.

LangChain is an open-source framework that makes building such applications dramatically easier. Think of it as a toolkit and orchestration layer for Large Language Models (LLMs). Instead of writing all the plumbing yourself, LangChain provides ready-made, composable components that you can snap together like LEGO bricks.

It was created by Harrison Chase in October 2022 and has become one of the most popular AI development frameworks in the world.

💡
Simple Analogy: LangChain is to AI what React is to web development — it doesn't replace the underlying engine (the LLM), but it gives you components, patterns, and structure to build sophisticated applications much faster and more reliably.

Why Use LangChain?

Without LangChain, connecting an LLM to external data sources, managing conversation history, orchestrating multiple steps, or building agents requires significant custom code. LangChain provides:

  • Composability — Connect models, prompts, tools, and memory like building blocks
  • Provider Agnostic — Switch between OpenAI, Google Gemini, Anthropic, Hugging Face, etc. with minimal code changes
  • Built-in Abstractions — Document loaders, text splitters, vector stores, retrievers — all ready to use
  • Agent Capabilities — Give your LLM the ability to use tools and make decisions
  • Active Community — Massive ecosystem of integrations and plugins

The LangChain Ecosystem

LangChain is not just a single library — it's an entire ecosystem of tools built around a common philosophy. Here are the four main members of the family:

Chat Models — Deep Dive

Chat models are the most commonly used type today. They work with messages — structured units of conversation with roles: SystemMessage, HumanMessage, and AIMessage.

Model Parameters

You can fine-tune model behavior with parameters passed during initialization:

  • temperature — Controls randomness. 0 = deterministic, 1 = creative (default varies)
  • max_output_tokens — Maximum tokens in the response
  • top_p — Nucleus sampling threshold
  • top_k — Limits to top-k token choices

Prompts & Templates

Hardcoding prompts in your application is brittle and hard to maintain. Prompt Templates are reusable, parameterized prompts — you define the structure once and fill in variables dynamically at runtime.

Think of them like Python f-strings, but with superpowers: validation, formatting, and composition.

PromptTemplate — Simple Text Prompts

Use PromptTemplate when working with simple text-based prompts (non-chat). Variables are wrapped in curly braces: {variable_name}.

⚡ General Syntax
template = PromptTemplate(input_variables=["<var1>", "<var2>"], template="... {var1} ... {var2} ...")
In [6] PromptTemplate Example
python 
from langchain_core.prompts import PromptTemplate

# Define a reusable template
# General Syntax:
# template = PromptTemplate(
#     input_variables=["", ""],
#     template="Your template text with {var1} and {var2}"
# )
template = PromptTemplate(
    input_variables=["topic", "audience"],
    template="Explain {topic} to a {audience} in 3 bullet points."
)

# Format the template with actual values
formatted_prompt = template.format(topic="neural networks", audience="10-year-old")
print(formatted_prompt)
Out [6]:
Explain neural networks to a 10-year-old in 3 bullet points.
In [7] Using PromptTemplate with a Model
python 
from langchain_google_genai import ChatGoogleGenerativeAI

ChatPromptTemplate — For Chat Models

ChatPromptTemplate is the preferred way to build prompts for chat models. It lets you define multiple messages with different roles: system, human, and ai.

⚡ General Syntax

    ("human", "{user_input}")

Few-Shot Prompting

Few-shot prompting is the technique of giving the model a few examples of input-output pairs within the prompt itself, so it learns the pattern you want it to follow. This dramatically improves consistency and accuracy for structured tasks.

In [9] Few-Shot Prompting
python 
from langchain_google_genai import ChatGoogleGenerativeAI
from langchain_core.prompts import ChatPromptTemplate, FewShotChatMessagePromptTemplate

model = ChatGoogleGenerativeAI(model="gemini-3.0-flash", api_key=GOOGLE_API_KEY)

# Step 1: Define example input-output pairs
examples = [
    {"input": "happy",   "output": "sad"},
    {"input": "fast",    "output": "slow"},
    {"input": "bright",  "output": "dark"},
]

# Step 2: Create an example prompt template
example_prompt = ChatPromptTemplate.from_messages([
    ("human", "{input}"),
    ("ai", "{output}"),
])

# Step 3: Create FewShotChatMessagePromptTemplate
few_shot_prompt = FewShotChatMessagePromptTemplate(
    example_prompt=example_prompt,
    examples=examples,
)

# Step 4: Build the final prompt
final_prompt = ChatPromptTemplate.from_messages([
    ("system", "You find the antonym of the given word."),
    few_shot_prompt,
    ("human", "{input}"),  # New question
])

# Invoke
chain = final_prompt | model
result = chain.invoke({"input": "ancient"})
print(result.content)  # Should output: "modern" or similar
Out [9]:
modern

MessagesPlaceholder

MessagesPlaceholder is a special slot in a ChatPromptTemplate that is filled with a list of messages at runtime. This is essential for implementing chat history and memory.

In [10] MessagesPlaceholder for Chat History
python 
from langchain_google_genai import ChatGoogleGenerativeAI
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from langchain_core.messages import HumanMessage, AIMessage

model = ChatGoogleGenerativeAI(model="gemini-3.0-flash", api_key=GOOGLE_API_KEY)

# MessagesPlaceholder acts as a dynamic slot for a list of messages
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful assistant."),

])

# Simulate chat history

    HumanMessage(content="My name is Alex."),
    AIMessage(content="Hello Alex! How can I help you today?"),
]

chain = prompt | model
response = chain.invoke({
    "history": history,
    "input": "What is my name?"
})
print(response.content)

Output Parsers

By default, a model returns an AIMessage object. Output Parsers transform that raw response into a more useful format: a plain string, a JSON dict, a Python object, or a custom structure. They are the final step in a chain.

StrOutputParser

The simplest parser. Extracts the .content string from an AIMessage. Use this whenever you just need plain text.

In [11] StrOutputParser
python 
from langchain_google_genai import ChatGoogleGenerativeAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

model = ChatGoogleGenerativeAI(model="gemini-3.0-flash", api_key=GOOGLE_API_KEY)
parser = StrOutputParser()

# Build a chain using the pipe (|) operator — more on this in LCEL section
chain = model | parser

# Now invoke returns a plain string, not an AIMessage
result = chain.invoke("What is 2+2?")
print(type(result))  # <class 'str'>
print(result)        # "4"
Out [11]:
<class 'str'> 4

JsonOutputParser

Parses the model's response as JSON. You should instruct the model to output valid JSON in your prompt. The parser handles the string-to-dict conversion automatically.

In [12] JsonOutputParser Example

from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import JsonOutputParser

result = chain.invoke({"name": "Priya", "age": "28", "job": "data scientist"})

PydanticOutputParser

The most powerful parser. Uses Pydantic models to define an exact schema for the output. LangChain automatically tells the model what format to follow via format instructions, and validates the response against your schema.

In [13] Define Pydantic Schema
python 
from pydantic import BaseModel, Field
from typing import List

# Define your expected output schema
class BookReview(BaseModel):
    title: str = Field(description="The book title")
    author: str = Field(description="The author's name")
    rating: int = Field(description="Rating from 1 to 10", ge=1, le=10)
    summary: str = Field(description="A one-sentence summary")
    pros: List[str] = Field(description="List of positive points")
    cons: List[str] = Field(description="List of negative points")
In [14] Use PydanticOutputParser
python 
from langchain_google_genai import ChatGoogleGenerativeAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import PydanticOutputParser

model = ChatGoogleGenerativeAI(model="gemini-3.0-flash", api_key=GOOGLE_API_KEY)

# Create the parser with your Pydantic model
parser = PydanticOutputParser(pydantic_object=BookReview)

# The parser can generate format instructions automatically!
format_instructions = parser.get_format_instructions()

prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a book critic. {format_instructions}"),
    ("human", "Review the book: {book_title}")
])

chain = prompt | model | parser

result = chain.invoke({
    "book_title": "The Hitchhiker's Guide to the Galaxy",
    "format_instructions": format_instructions
})

print(type(result))        # <class 'BookReview'>
print(result.title)        # Typed attribute access
print(result.rating)       # 9
print(result.pros)         # ['Witty humor', 'Creative world-building', ...]

LCEL & Chains

LCEL — LangChain Expression Language — is the modern way to compose chains in LangChain. It uses the pipe operator (|) to connect components. Think of it as Unix pipes, but for AI components.

The core idea: every component in LangChain is a Runnable, meaning it has .invoke(), .stream(), .batch(), and .ainvoke() methods. The pipe operator connects Runnables, passing the output of one as input to the next.

PromptTemplate
|
ChatModel
|
OutputParser
→
Result
In [15] LCEL — The Pipe Operator
python 
from langchain_google_genai import ChatGoogleGenerativeAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

model = ChatGoogleGenerativeAI(model="gemini-3.0-flash", api_key=GOOGLE_API_KEY)

# Build a chain using | (pipe operator)
# Data flows left to right: prompt → model → parser
chain = (
    ChatPromptTemplate.from_messages([
        ("system", "You are a Python code generator. Return only code, no explanation."),
        ("human", "Write a function to {task}")
    ])
    | model
    | StrOutputParser()
)

result = chain.invoke({"task": "calculate the factorial of a number"})
print(result)
Out [15]:
def factorial(n): if n == 0 or n == 1: return 1 return n * factorial(n - 1)

RunnablePassthrough & RunnableLambda

RunnablePassthrough passes data through unchanged — useful for preserving original input while other parts transform it. RunnableLambda wraps any Python function as a Runnable.

In [16] RunnablePassthrough and RunnableLambda
python 
from langchain_core.runnables import RunnablePassthrough, RunnableLambda
from langchain_google_genai import ChatGoogleGenerativeAI
from langchain_core.output_parsers import StrOutputParser

model = ChatGoogleGenerativeAI(model="gemini-3.0-flash", api_key=GOOGLE_API_KEY)

# RunnableLambda wraps any function
def add_exclamation(text: str) -> str:
    return text + "!!!"

# Chain: model → parser → custom function
chain = model | StrOutputParser() | RunnableLambda(add_exclamation)
result = chain.invoke("Say hello in one word")
print(result)  # "Hello!!!"

# RunnablePassthrough — passes input unchanged
passthrough = RunnablePassthrough()
print(passthrough.invoke({"key": "value"}))  # {"key": "value"}

RunnableParallel — Run Chains in Parallel

RunnableParallel runs multiple chains simultaneously and merges their outputs into a dictionary. This is perfect for generating multiple perspectives or transformations at once.

In [17] RunnableParallel Example
python 
from langchain_google_genai import ChatGoogleGenerativeAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnableParallel, RunnablePassthrough

model = ChatGoogleGenerativeAI(model="gemini-3.0-flash", api_key=GOOGLE_API_KEY)
parser = StrOutputParser()

# Two independent chains for the same topic
pro_chain = (
    ChatPromptTemplate.from_template("List 3 pros of {topic}.")
    | model | parser
)
con_chain = (
    ChatPromptTemplate.from_template("List 3 cons of {topic}.")
    | model | parser
)

# Run both IN PARALLEL — much faster than sequential
parallel_chain = RunnableParallel(
    pros=pro_chain,
    cons=con_chain,
    topic=RunnablePassthrough()  # Keep original input too
)

result = parallel_chain.invoke({"topic": "remote work"})
print("PROS:\n", result["pros"])
print("\nCONS:\n", result["cons"])
print("\nOriginal Input:", result["topic"])

Sequential Chains

A sequential chain takes the output of one step and feeds it directly as input to the next. This is the default behavior of the | operator — each step's output becomes the next step's input.

In [18] Multi-Step Sequential Chain
python 
from langchain_google_genai import ChatGoogleGenerativeAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

model = ChatGoogleGenerativeAI(model="gemini-3.0-flash", api_key=GOOGLE_API_KEY)
parser = StrOutputParser()

# Chain 1: Generate code
code_chain = (
    ChatPromptTemplate.from_template("Write a Python function to {task}. Return only the code.")
    | model | parser
)

# Chain 2: Review the generated code
review_chain = (
    ChatPromptTemplate.from_template(
        "Review this Python code for bugs and improvements:\n\n{code}\n\nProvide a brief review."
    )
    | model | parser
)

# Connect them: output of code_chain feeds into review_chain as {code}
full_chain = code_chain | (lambda code: {"code": code}) | review_chain

result = full_chain.invoke({"task": "reverse a linked list"})
print(result)

Memory & Chat History

LLMs are stateless by design — they remember nothing between calls. Each request is processed in isolation. This is a fundamental limitation when building conversational applications.

Memory in LangChain solves this by maintaining a record of the conversation (or a summary of it) and injecting it into each new request. LangChain provides several memory strategies:

  • ConversationBufferMemory — Stores the entire conversation verbatim
  • ConversationBufferWindowMemory — Stores only the last N exchanges
  • ConversationSummaryMemory — Summarizes older messages to save tokens
  • ConversationSummaryBufferMemory — Hybrid: recent messages verbatim, older ones summarized

ConversationBufferMemory

In [19] Manual Chat History Pattern (LCEL-compatible)
python 
from langchain_google_genai import ChatGoogleGenerativeAI
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from langchain_core.messages import HumanMessage, AIMessage
from langchain_core.output_parsers import StrOutputParser

model = ChatGoogleGenerativeAI(model="gemini-3.0-flash", api_key=GOOGLE_API_KEY)

prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a friendly AI assistant with a great memory."),
    MessagesPlaceholder(variable_name="history"),
    ("human", "{input}"),
])

chain = prompt | model | StrOutputParser()

# Maintain history manually
history = []

def chat(user_input: str) -> str:
    response = chain.invoke({"input": user_input, "history": history})
    # Append to history after each turn
    history.append(HumanMessage(content=user_input))
    history.append(AIMessage(content=response))
    return response

# Multi-turn conversation
print(chat("My name is Arjun and I love Python."))
print(chat("What programming language do I love?"))  # Should remember Python
print(chat("And what's my name?"))                   # Should remember Arjun
Out [19]:
Hi Arjun! That's great — Python is a fantastic language! What do you like most about it? You mentioned that you love Python! Your name is Arjun!

ConversationSummaryMemory

For long conversations, storing everything becomes expensive (too many tokens). Summary memory uses the LLM itself to periodically summarize older messages, keeping only a compact summary instead of the full history.

In [20] ConversationSummaryBufferMemory
python 
from langchain.memory import ConversationSummaryBufferMemory
from langchain_google_genai import ChatGoogleGenerativeAI

model = ChatGoogleGenerativeAI(model="gemini-3.0-flash", api_key=GOOGLE_API_KEY)

# max_token_limit: When history exceeds this, older messages are summarized
memory = ConversationSummaryBufferMemory(
    llm=model,
    max_token_limit=200,
    return_messages=True
)

# Add messages
memory.save_context(
    {"input": "I'm building a web app in Django."},
    {"output": "That's great! Django is a powerful Python web framework."}
)
memory.save_context(
    {"input": "I need to integrate payments."},
    {"output": "I'd recommend Stripe for payment integration with Django."}
)

# Load the memory — it automatically summarizes when needed
memory_vars = memory.load_memory_variables({})
print(memory_vars)

RunnableWithMessageHistory

The modern, LCEL-compatible way to add persistent memory to any chain. It wraps a chain with automatic history loading and saving per session.

In [21] RunnableWithMessageHistory
python 
from langchain_google_genai import ChatGoogleGenerativeAI
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from langchain_core.output_parsers import StrOutputParser
from langchain_core.chat_history import BaseChatMessageHistory
from langchain_community.chat_message_histories import ChatMessageHistory
from langchain_core.runnables.history import RunnableWithMessageHistory

model = ChatGoogleGenerativeAI(model="gemini-3.0-flash", api_key=GOOGLE_API_KEY)

# In-memory store (use Redis or DB in production)
store = {}

Agents

An Agent is an LLM that can use tools, reason about which tools to use, act on the results, and repeat this process until it has solved the problem. Unlike a chain (which follows a fixed sequence), an agent makes dynamic decisions at runtime about what to do next.

This is the key distinction:

  • Chain: Predefined, fixed sequence of steps. You (the developer) decide what happens and in what order.
  • Agent: Dynamic, adaptive reasoning loop. The LLM decides which tools to call, in what order, based on the current situation.

The ReAct Pattern

ReAct (Reasoning + Acting) is the most common agent pattern. The agent cycles through:

  1. Thought — The agent reasons about what to do next ("I need to search for current weather in Tokyo...")
  2. Action — The agent calls a tool with specific inputs
  3. Observation — The agent sees the tool's output
  4. Repeat — Until the agent has enough information to give a final answer

LangGraph

LangGraph is a library for building stateful, multi-actor applications with LLMs. While LangChain's AgentExecutor is great for simple agents, LangGraph gives you much more control when you need:

  • Cycles — The agent can loop back and try again
  • Branching — Different paths based on conditions
  • Persistence — Save and resume state between calls
  • Human-in-the-loop — Pause execution and wait for human approval
  • Multi-agent coordination — Multiple agents working together

LangGraph models your application as a directed graph where:

  • Nodes are functions (actions, LLM calls, tool executions)
  • Edges are the connections between nodes (what runs next)
  • State is a shared data structure passed between nodes

StateGraph — Core Concept

The StateGraph is the main class in LangGraph. You define a State (a TypedDict), add nodes, add edges, and then compile the graph.

⚡ General Syntax
from langgraph.graph import StateGraph, START, END
from typing import TypedDict

class State(TypedDict):
    <field>: <type>

graph = StateGraph(State)
graph.add_node("<node_name>", <function>)
graph.add_edge(START, "<node_name>")
graph.add_edge("<node_name>", END)
app = graph.compile()

class RoutingState(TypedDict):

    category: str
    response: str

        ChatPromptTemplate.from_template(
            "Classify this as exactly one word — 'math', 'writing', or 'general':\n{input}"
        ) | model | StrOutputParser()

    response = model.invoke(f"Help with this writing task: {state['user_input']}").content
    return {"response": f"[WRITING EXPERT]: {response}"}

def general_assistant(state: RoutingState) -> RoutingState:
    response = model.invoke(state["user_input"]).content
    return {"response": f"[GENERAL]: {response}"}

def route_by_category(state: RoutingState) -> str:

graph.add_node("writing", writing_expert)

Multi-Agent Systems

A Multi-Agent System (MAS) is a system where multiple AI agents work together, each with specialized roles and capabilities, to solve complex tasks that would be too large or complex for a single agent.

Why use multiple agents?

  • Specialization — Each agent excels at its specific domain (research, coding, writing, etc.)
  • Parallelism — Agents can work on different parts of a problem simultaneously
  • Scalability — Add more agents without changing the core architecture
  • Context management — Each agent maintains a focused, smaller context window

The Supervisor Pattern

In the Supervisor Pattern, a central Supervisor agent receives the user's request, delegates subtasks to specialized Worker agents, collects their outputs, and synthesizes a final response. This is the most common multi-agent pattern.

→

In [38] Supervisor Agent with LangGraph
python 
from langgraph.graph import StateGraph, START, END
from langchain_google_genai import ChatGoogleGenerativeAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from typing import TypedDict, List, Literal

model = ChatGoogleGenerativeAI(model="gemini-3.0-flash", api_key=GOOGLE_API_KEY)
parser = StrOutputParser()

# ─── State ────────────────────────────────────────────────
class MASState(TypedDict):
    user_request: str
    next_agent: str
    research_output: str
    code_output: str
    final_response: str

# ─── Specialized Worker Agents ────────────────────────────
def researcher_agent(state: MASState) -> MASState:
    """Agent specialized in research and information gathering."""
    chain = (
        ChatPromptTemplate.from_template(
            "You are a research specialist. Research and summarize key information about:\n{topic}"
        ) | model | parser
    )
    result = chain.invoke({"topic": state["user_request"]})
    return {"research_output": result}

def coder_agent(state: MASState) -> MASState:
    """Agent specialized in writing Python code."""
    chain = (
        ChatPromptTemplate.from_template(
            "You are a Python expert. Write clean, working Python code for:\n{task}\n"
            "Context from research: {context}"
        ) | model | parser
    )
    result = chain.invoke({
        "task": state["user_request"],
        "context": state.get("research_output", "No research yet")
    })
    return {"code_output": result}

def supervisor_agent(state: MASState) -> MASState:
    """Supervisor decides which agent to call next."""
    chain = (
        ChatPromptTemplate.from_template(
            """You are a supervisor managing specialized agents.
            
User request: {request}
Research done: {has_research}
Code done: {has_code}

Respond with EXACTLY ONE of: 'researcher', 'coder', 'FINISH'
- 'researcher': if research/information is needed
- 'coder': if code implementation is needed
- 'FINISH': if we have enough information to answer

Your decision:"""
        ) | model | parser
    )
    decision = chain.invoke({
        "request": state["user_request"],
        "has_research": bool(state.get("research_output")),
        "has_code": bool(state.get("code_output"))
    }).strip().lower()
    
    if "finish" in decision: return {"next_agent": "FINISH"}
    if "coder" in decision:  return {"next_agent": "coder"}
    return {"next_agent": "researcher"}

def synthesizer_node(state: MASState) -> MASState:
    """Combines all agent outputs into a final response."""
    chain = (
        ChatPromptTemplate.from_template(
            """Synthesize this information into a comprehensive final answer.
            
Original request: {request}
Research findings: {research}
Code solution: {code}

Provide a clear, complete response:"""
        ) | model | parser
    )
    final = chain.invoke({
        "request": state["user_request"],
        "research": state.get("research_output", "N/A"),
        "code": state.get("code_output", "N/A")
    })
    return {"final_response": final}

# ─── Route based on supervisor decision ───────────────────
def route_to_agent(state: MASState) -> str:
    return state.get("next_agent", "researcher")

# ─── Build Multi-Agent Graph ──────────────────────────────
graph = StateGraph(MASState)
graph.add_node("supervisor", supervisor_agent)
graph.add_node("researcher", researcher_agent)
graph.add_node("coder", coder_agent)
graph.add_node("synthesizer", synthesizer_node)

graph.add_edge(START, "supervisor")
graph.add_conditional_edges("supervisor", route_to_agent, {
    "researcher": "researcher",
    "coder": "coder",
    "FINISH": "synthesizer",
})
graph.add_edge("researcher", "supervisor")  # Back to supervisor
graph.add_edge("coder", "supervisor")       # Back to supervisor
graph.add_edge("synthesizer", END)

mas_app = graph.compile()

# Run the multi-agent system
result = mas_app.invoke({
    "user_request": "How do I implement a binary search tree in Python? "
                    "Include explanation and working code.",
    "next_agent": "",
    "research_output": "",
    "code_output": "",
    "final_response": ""
})

print("═" * 50)
print("FINAL RESPONSE:")
print("═" * 50)
print(result["final_response"])

    data_cleaned: str
    insights: str

def data_cleaner(state: AnalysisState) -> AnalysisState:

    ) | model | parser
    return {"insights": chain.invoke({"task": state["task"], "cleaned": state["data_cleaned"]})}

    sub_result = analysis_subgraph.invoke({"task": state["user_query"], "data_cleaned": "", "insights": ""})
    return {"analysis_result": sub_result["insights"]}

top_graph.add_node("analysis_team", run_analysis_team)
top_graph.add_node("writer", final_writer)
top_graph.add_edge(START, "analysis_team")
top_graph.add_edge("analysis_team", "writer")
top_graph.add_edge("writer", END)

# Shared state accessible by all agents
class CollaborativeState(TypedDict):
    messages: Annotated[List, operator.add]
    shared_notes: str  # A shared "whiteboard" all agents can read/write
    current_agent: str

# Agent A: Planner
def planner_agent(state: CollaborativeState) -> CollaborativeState:
    system = SystemMessage(content=
        f"You are the Planner agent. Your role is to break down tasks into steps. "
        f"Current shared notes:\n{state.get('shared_notes', 'Empty')}"
    )

    return {
        "messages": [AIMessage(content=f"[PLANNER]: {response.content}")],
        "shared_notes": notes,
        "current_agent": "executor"
    }

# Agent B: Executor
def executor_agent(state: CollaborativeState) -> CollaborativeState:
    system = SystemMessage(content=
        f"You are the Executor agent. Your role is to implement the plan. "
        f"Current shared notes:\n{state.get('shared_notes', 'Empty')}"
    )
    response = model.invoke([system] + state["messages"])
    notes = state.get("shared_notes", "") + f"\n[EXECUTOR]: {response.content[:200]}"
    return {
        "messages": [AIMessage(content=f"[EXECUTOR]: {response.content}")],
        "shared_notes": notes,
        "current_agent": "done"
    }

def route_agents(state: CollaborativeState) -> str:
    return state.get("current_agent", "planner")

collab_graph.add_node("executor", executor_agent)

collab_graph.add_conditional_edges("planner", route_agents, {
    "executor": "executor", "done": END
})
collab_graph.add_conditional_edges("executor", route_agents, {
    "planner": "planner", "done": END
})

import time
from typing import List

def rate_limited_batch(chain, inputs: List[dict], delay: float = 0.5) -> List:
    """Process inputs with a delay between each call to respect rate limits."""
    results = []
    for i, inp in enumerate(inputs):
        result = chain.invoke(inp)
        results.append(result)
        if i < len(inputs) - 1:  # Don't sleep after the last item
            time.sleep(delay)
    return results
